## Built (V0 substrate)

- **V0.3.1 — post-release hardening (2026-07-14).** `llmcore` `cache_control` markers on the shared system-prompt seam — every LLM call in the fleet (arc/router/expander/summarizer/classifier/asker/judge) now caches its tools+system prefix server-side; ~10% cost on repeat prefixes, disk cache key unchanged so existing `.cache/*.json` still hit. `bin/saturday-stack` gains `doctor` (pre-launch venv health check) and `repair-audio` (one-word rebuild) with a Python-version drift detector — the audio pane no longer dies silently after an OS Python upgrade. `bin/saturday-claude` gains a `cc-<basename>` collision guard: two projects with the same leaf dir (e.g. `.../dogfood/web` and `.../firecrawl/web`) no longer silently hijack each other's tmux session. `watcher/decodeProjectName` handles `~` and `~/Documents` cases that were leaking raw encoded paths into arc labels.
- **V0.2.7 — Sonnet expander, mic RMS, mayor color register, voice callsigns, effigy.** ExpanderModel upgraded to `claude-sonnet-4-6` (was Haiku); router and a new `SummarizerModel` constant stay on Haiku. Hotphrase opt-in already accepts the latency tax, so trade ms for fewer over-cautious asks. Eval pass-rate dropped 77% → 63% but failure profile shifted from `over_cautious` to `over_eager` — recoverable mistakes beat silent asks. An attempted "expand-v4" Haiku prompt with appended rules regressed 77% → 57%; reverted, lesson recorded in `expander.go` + memory. New audio→mayor frame `{"type":"rms","db":...}` at 5 Hz; mayor mirrors a 32-sample dBFS ring into `MayorState.Rms`; `saturday-thinking` renders a sparkline in the status card (mute = flat, talking = peaks). Mayor stderr now uses a three-act color register: cyan `← utt` (heard), dim cyan `→ route:`, bright yellow `→ Saturday → <project>` (sent), magenta `? expander asks`, red `✗ expander declined`, green `✓ completion report`, bright red `× err`. The `bin/` launcher scripts (`saturday-stack`, `saturday-claude`) are kept symlinked from `$(go env GOPATH)/bin/` to the repo so script edits propagate without a manual reinstall. Eval re-run with v3 prompts captured fresh baseline before model swap. Voice-friendly callsigns now appended to every expand-mode inject as a `[saturday: ...]` rule asking CC to label enumerated items with phonetic callsigns (alpha/bravo/cherry/...) so the user can later say "the bravo one"; verbatim mode skipped (the inject is the literal text). Half-2 of callsigns (expander prompt resolves callsign references in utterances) deferred until callsigns appear in real session JSONL — needs corpus evidence to know what referent patterns to handle. `saturday.effigy` written and wired in (relocated to `llmcore/saturday.effigy`, embedded via `go:embed`, appended to ExpanderSystem and SummarizerSystem with cache prefixes `expand-v3-sonnet-effigy` / `summary-v2`). Eval pass-rate moved 63% → **80%** with the persona file feeding the expander — over_cautious dropped from 5 to 1 because the voice register's "Don't seek validation" rules push toward inject more reliably than the original prompt's bias rules. `llmcore/verifier.go` (V0.2.7 verifier) lexically strips NEVER-list violations from summarizer output (apologies, hedges, preamble, exclamation, performed feeling) — cheap safety net, no extra LLM call. Influences acknowledged in the effigy header: Jarvis pattern (existing), Banks-Mind register, TARS, Bioshock-derived hotphrase.
- `llmcore/` — shared Go package: router/expander prompts + tools, content-hash cache, Anthropic API plumbing, dotenv loader, `State`/`ToolUseSnap` types. Single source of truth across mayor + both evals; cache-key compatible with pre-lift binaries (verified: zero new cache files post-lift).
- `eval/expander_backtest.go` — Haiku-as-expander, Sonnet-as-judge, **0% destructive** on local corpus; pass-rate ~50% (lots of `over_cautious` and `needs_ask_correct`).
- `eval/router/main.go` — Haiku-as-router on `(utterance, N candidates)`, **93% routing accuracy / 4 candidates** on synthetic corpus.
- **V0.1.4 — direct-write inject path for big sessions.** Mayor estimates post-compact tokens (bytes since last `isCompactSummary` ÷ 4); above `--inject-direct-threshold` (default 80k), it skips `claude --resume --print` and appends the user turn directly to the target JSONL under `flock`, with full schema (`parentUuid` chained off latest `last-prompt`'s `leafUuid`, `promptId`, `permissionMode`, etc., empirically captured from CC 2.1.126). Sync hook frames such turns as `(no auto reply — request still pending)` so the live pane reliably acts on next interaction. Eliminates autocompact-divert in big sessions; preserves headless autonomous-background-work in small ones. Probe-verified: a directly-written user turn loaded into context cleanly on `claude --resume`, the model answered based on the dangling turn's content.
- **V0.2.4 — proactive completion reports (Phase 3 of the Jarvis arc).** After a successful tmux or direct-write inject, mayor stamps the target session into `pendingInjects[sessionID]` with the original utterance + the JSONL size at inject time. A polling goroutine ticks every 3 s; when the JSONL has been size-stable for 5 s AND the latest assistant block is `text` (not `tool_use`/`thinking`), mayor pulls `state.last_assistant_text` from the watcher, calls a new `llm.RunSummarize` (Haiku, ≤15-word past-tense report, cache prefix `summary-v1`), and writes `{"type":"speak","text":...}` over the audio sock. User gets *"lucida tests done — 32 passed, 2 failed"* without asking. Filters: 200-byte trivial-growth threshold (one-line bash → no report; the stock ack already played), 5 s minimum-elapsed (instant tasks → silent), 10 min TTL (drop stale tracker entries). The detection heuristic was empirically calibrated by writing `bin/saturday-observe` first — a stdlib-Python probe that tails active CC JSONLs and prints classified per-block events with inter-event timing, validating the assumption that each text/tool_use/thinking block is its own JSONL line and that intra-chain inter-event gaps are <2 s. Audio-sock writes now serialize through a new `audioMu` mutex since the polling goroutine and the synchronous pipeline both write concurrently. `bin/saturday-observe` ships as a debug tool — useful when calibrating future detector tweaks or diagnosing why a completion didn't fire.
- **V0.2.3 — XDG file layout + `saturday-stack` launcher.** Mayor's `.env` and `.cache/` migrated out of repo dirs to XDG-standard paths (`$XDG_CONFIG_HOME/saturday/config`, `$XDG_CACHE_HOME/saturday/llm/`). Legacy `<scriptDir>/.env` fallback preserved for in-repo dev. `bin/saturday-stack` brings up watcher / mayor / audio in a 3-pane tmux session named `saturday-stack`, focuses the audio pane (where SPACEBAR-mute lives), `remain-on-exit on` so dead panes stay readable. Subcommands: `saturday-stack stop|down|kill` cleanly tears down via SIGHUP, `saturday-stack status` lists pane PIDs and current commands. First-run auto-migration: `saturday-mayor/.env` → `~/.config/saturday/config`. Pre-flight checks for venv + binaries before constructing the session.
- **V0.2.2 — Jarvis-pattern voice loop (verbatim default + hotphrase + stock acks).** Wire format gained a `mode` field: `"verbatim"` (default) — utterance text becomes the inject directly, no expander LLM, no narration; `"expand"` — full router+expander pipeline plus narration TTS. Hotphrase prefix `"would you kindly"` (configurable via `--hotphrase`) opts an utterance into expand mode. Sidecar pre-renders 5 stock acknowledgment phrases via Kokoro on startup ("On it.", "Got it.", "Right away.", "Sure thing.", "Doing that now.") and plays a random pick instantly on every utterance — instant audible feedback. Voice blending: `--tts-voice am_adam+af_sarah` averages embeddings. STT speedups: `beam_size=5→1` (free), VAD silence-flush 700→500ms. Vocab handling: terms split on `-`/`_`, alphanumeric-only, period-separated in initial_prompt, capped at 20 (was 60) — kills the over-biasing pattern that caused Whisper to ad-hoc hyphenate unrelated word pairs. `huggingface_hub.try_to_load_from_cache` suppresses repeat unauthenticated-request warnings. Kokoro models pulled from `github.com/thewh1teagle/kokoro-onnx` releases (HF onnx-community variant lacks single voices.bin), cached at `~/.cache/saturday-audio/kokoro/`.
- **V0.2.1 — TTS reply on `action="ask"` (closes the dialog loop).** Mayor's audio socket is now bidirectional: when the expander asks a clarifier, mayor writes `{"type":"speak","text":...}` back over the existing sidecar conn. Sidecar's `conn_manager` (single-owner thread, drains `outbox_q` for utterances out + fills `event_q` from incoming events) hands the speak event to a `tts_player` thread that synthesizes via Kokoro (24 kHz, default voice `af_heart`) and plays through the default output device. `state.tts_playing` event auto-suppresses STT during playback (mic captures TTS audio otherwise → feedback loop); VAD speech-onset shows `…tts` in that window so the indicator distinguishes user-speech-during-tts from normal hearing. Also: V2 expander prompt — explicit "do NOT pre-resolve the answer using state" rule with right-vs-wrong example; cache prefix bumped from `expand-baseline` → `expand-v2` so old cache silently invalidates rather than serving the wrong shape (eval baseline pass-rate now stale until re-run). Mayor cache LRU added: `--cache-max` flag (default 1000 files), startup prune drops oldest by mtime — bounds growth from open-mic's continuous-utterance accumulation.
- **V0.2.0 — audio sidecar (open-mic).** `saturday-audio/main.py` Python sidecar: PortAudio mic capture at 16 kHz mono, Silero VAD utterance segmentation (700 ms silence flush), faster-whisper `small.en` int8 STT on CPU. Emits line-delimited JSON (`{"type":"utterance","text":...,"ts":...}`) over a Unix socket to mayor. Mayor's `--audio-sock <path>` flag swaps the stdin reader for a socket listener; empty (default) keeps stdin mode for testing. Open-mic by default; SPACEBAR in the sidecar's terminal toggles `[live]` / `[muted]` (mute is a hard cutoff before STT — no logging, no emit). `q` quits. Daily transcript log at `~/.claude/saturday/transcripts/YYYY-MM-DD.log`. Wire format extensible: unknown `type` values ignored. `ptt` mode reserved as flag-stub for V0.2.1; smart-turn classifier and TTS reply also deferred.
- **V0.1.5 — tmux send-keys as primary inject path.** When the target session's claude is running inside a tmux pane (discovered by walking `tmux list-panes -aF '#{pane_id} #{pane_pid}'` and descending each pane's `/proc` tree to match cwd), mayor types the expanded prompt + Enter into that pane via `tmux send-keys -l <text>` + `tmux send-keys Enter`. Live claude executes natively — full scrollback, permissions inherit, no autocompact-divert, no orphan dance, no cross-session typing tax. Direct-write and headless paths become fallbacks for non-tmux sessions. Adoption: `bin/saturday-claude` wrapper starts (or attaches to) a per-project tmux session and forwards all CLI args to claude. Naming: tmux session name = `cc-<basename PWD>`, attached via `tmux new-session -A -s` so re-running from the same dir reattaches deterministically. One of the misses is conf=0.95 — a wrong-confident-route — and a V0.1-style abort gate couldn't have caught it in time anyway. Mitigation moves to V0.2 (audio false-positive gating) or to better routing.

## V0.1 — finish the silent-loop minus voice

- **Injection** — `claude --resume <sid> --print '<text>'` headless. No tmux, no keystroke injection, terminal-emulator-agnostic. Probed live; works clean. See `INJECTION.md` for rationale and probe data.
- **Sync hook** — `UserPromptSubmit` hook that diffs the session JSONL against a per-session cursor and re-injects out-of-band turn context as `<earlier-turn channel="voice">` so the live pane sees Saturday's headless inserts on the user's next interaction. Filters meta turns (`isMeta`, `isCompactSummary`) and slash-command framing (`<command-name>`, `<local-command-stdout>`) to keep voice context clean across `/compact`. Frozen output template for cache stability.
- **Mayor** — `saturday-mayor`, the V0.1 silent-loop orchestrator. Reads stdin → watcher socket query → router (Haiku) → expander (Haiku) → `claude --resume --print` headless. Single mayor process, sequential pipeline. Router and expander prompts mirror `eval/` until shared `llmcore` package lands.
- **No mediator in V0.1.** Originally specced as a ~1s text-mode confirmation gate — cut because the window is too short to be a real abort and there's no false-positive risk in V0.1's stdin-as-STT mode (user authored every input). Mayor execs `claude --resume --print` directly and prints a one-line proposal log to stderr. Real confirmation mechanism (press-to-commit, not press-to-abort) returns in V0.2 if audio surfaces ambiguous transcripts.

## V0.2 — speech in the loop

- `audio/` — Python sidecar: faster-whisper STT (200ms target) + Kokoro TTS for meta-notifications, on a Unix socket. Local STT is for privacy, not performance.
- VAD-driven open-mic with a layered saliency gate (rule pass + small LLM for borderline cases).
- Wire format: `utterance → router → expander → confirmation gate (if needed) → injection`. Each stage testable in isolation; harness covers each. The gate becomes real only if VAD/saliency false-positives demand it.

## Medium-term

- **Per-session lockfile coordination** — `flock`-based exclusive locks at `~/.claude/saturday/locks/<sid>` taken by both mayor (around `claude --resume --print` exec) and a paired hook in the user's interactive CC (acquire on `UserPromptSubmit`, release on `Stop`). Replaces V0.1.1's quiet-window probe with deterministic mutual exclusion against the user's live writes. Cost: two new hooks, a small state machine, and the assumption that every CC instance the user might type into has the hook installed. Trigger: the quiet-window probe demonstrably misses real collisions, or audio-mode V0.2 produces enough injects per minute that probabilistic mitigation isn't enough.
- ~~**Slow-loop summarizer**~~ — landed as the V0.2.7 arc (`saturday-mayor/main.go::runArcRefresher`, `llmcore/arc.go`, 5-min cadence per active session, Haiku, content-hash cached so unchanged sessions cost nothing). Fed into the expander/router state as `session_arc`. Original spec below preserved for design context.
- **Status indicator (das blinkenlights)** — terminal-corner tag rendered into the *target* CC pane (not just mayor's own terminal) while mayor is processing. Phases: `[heard]` → `[routing]` → `[expanding]` → `[injecting → <project>]` → fading `[done]`. Two reasons it's load-bearing, not cosmetic: (1) in V0.2 audio mode the source utterance isn't visibly typed, so the user has no feedback loop without it. (2) **More fundamental:** when mayor injects via `claude --resume --print`, the target pane's running claude process can't be triggered to refresh — CC is a closed box, only re-reads JSONL on its own turn boundaries. The user sees nothing in the target pane until they next press Enter (which fires the sync hook and surfaces the inject as `<earlier-turn channel="voice">`). Blinkenlights closes that gap by overlaying a status tag *over* the live pane via `/dev/tty` + ANSI absolute-positioning, independent of CC's render loop — the tag lives outside Ink's managed frame, so CC's repaints don't erase it. Mayor resolves the target's claude PID → tty via `/proc/<pid>/fd/0` (or has watcher track it). Trigger: pulled forward — needed even in silent-mode given the target-pane invisibility gap.
- **Live turn detection (cribbed)** — pull LiveKit's open-weights turn-detector or Pipecat's smart-turn into the V0.2 audio sidecar as a single Python call. End-of-utterance from transcript semantics, not just acoustic silence — catches the "uhh let me think" pause that VAD-silence-timeout fumbles. Don't adopt the framework; copy the model. ~2025 saw a wave of OSS in this space (Pipecat, LiveKit Agents, TEN); Saturday's architecture isn't a chat-completion agent loop, so framework adoption is wrong scope — just borrow the smart-turn classifier. Trigger: V0.2 audio false-positives (premature send on hesitation, or trailing speech captured as a second utterance).
- **Dialog management (state-of-2 clarifier)** — when router or expander conf falls below threshold and the audio sidecar exists, Saturday speaks one rising-intonation candidate (`"hindcast?"`); user replies "yes" / "no" / a project name within ~3s; anything else → fall back to silent (drop the utterance). One round, no recursion, fixed answer vocabulary. Keeps disambiguation in the same channel as the original utterance (banter stays live and fluid; only the action — the inject — is async). Constraint: hold the original utterance in mayor's queue for the duration of the clarifier window. Avoids reinventing turn-taking from scratch by deliberately keeping the protocol state-of-2 not open-dialog. Trigger: V0.2 audio working + lossy-disambiguation observed in practice (utterances dropped because `--conf-threshold` gate fired too often).
- **Voice-friendly callsigns (referent grounding)** — when the expander injects into a target session whose response will plausibly enumerate >1 peer items, the inject prefixes a system-prompt rule: "label each item with a phonetically-distinct callsign from a fixed lexicon (NATO alpha/bravo/... or pet-name style: apollo/bear/cloud/dust/ember/...). Reuse callsigns within this session for the same items. Don't label single-item or pure-prose responses." Then voice utterances like `"fix the bear one"` route to the session whose recent `last_assistant_text` contains the `bear` callsign and the expander expands the reference. Labels live in the JSONL — the user reads them on screen, watcher exposes them via `last_assistant_text`, no separate registry. Prior art: NATO ICAO phonetic alphabet (1956, robust under low-fidelity channels and accents), Vimium's link-hint system (keyboard analog), IBM/Nuance "mark-and-recall" patterns. Tradeoffs: lexicon ≤ ~26 covers 99% of real cases; phonetic distinctness > brevity (Whisper handles "alpha" better than "A"); avoid callsigns that collide with the working domain (don't use `git` as a callsign while doing git work). Trigger: V0.2 audio mode + observed referent friction (user trying to say "the third one" or rereading long file paths aloud).
- ~~**Stack startup ergonomics**~~ — landed as `bin/saturday-stack` (V0.2.3): dedicated tmux session with mayor + thinking + audio panes, `stop`/`status`/`doctor`/`repair-audio` subcommands, XDG-standard config/cache paths. Watcher runs headless in the background from V0.2.6 forward. Design discussion below preserved.
- **Drivermap integration** — real-time mechanism profile from [drivermap](https://github.com/justinstimatze/drivermap) fed into expander prompt for state-aware register adaptation (less flag-flipping when the user is in flow_state, more questions when in ambiguity_zone).
- **Effigy-driven verifier** — post-generation check that expansions match persona NEVER list. Runs cheap (small LLM), abstains on ambiguity.
- **Expansion-feedback loop** — track which injected prompts the user edited vs accepted post-inject. Tunes the inject-vs-ask threshold per utterance pattern over time.
- **Ask-mode deep context (recent JSONL pull)** — when Saturday's ask-mode answers questions about another session, arcs cover the broad theme but miss specifics ("what was the exact error in lucida tests", "what file did slimemold just touch"). Defer: pull recent JSONL excerpts from the queried session on the fly — last N blocks, or content-filtered (tool_result containing a keyword from the question), summarize down, fold into the ask-response prompt. Cost is non-trivial: 10s of MB read per ask, extraction latency, a second LLM summarize step to fit response context. Trigger: ask-mode (V0.3) lands, observed gap where arcs alone consistently miss the user's actual question. Until then ask-mode answers from arcs + recent state only; specifics get a "Saturday only knows the arc — go look".
- **Federated state** — Tailscale-synced watcher socket across desktop / laptop / phone. Saturday's state follows the user.
- **Roaming voice** — Aria Me BT headset, mSBC capture, wake-word-or-saliency open-mic. Was descoped in V0.2 because of the BT codec ceiling and warmup latency.

## Speculative / later

- Breakroom modes — Saturday detects fatigue via [drivermap](https://github.com/justinstimatze/drivermap), suggests a quick chess/backgammon round in a side pane.
- Long-arc tracking — drivermap evolving across weeks/months, surfacing patterns that go unnoticed in the moment.
- Register tuning — "vast capability worn casually." Probably the hardest part. Effigy persona + careful expander prompt + post-generation verifier all working together.
- Substrate-presence — Saturday running on every device, with a coherent shared model. Likely needs a pseudo-CRDT for the drivermap profile to merge across devices.

- **3D-desktop substrate (deferred).** Saturday is the mouth/hands; [lucida](https://github.com/justinstimatze/lucida) is the peripheral state-flow indicator (das blinkenlights, intentionally ephemeral); a 3D-desktop substrate (Immersed-like — pattern, not the company) carries the actual terminal/editor windows. Lucida's `mixed3d` mode is already structured for WebXR — the cell-to-canvas rasterizer + `WebXRManager` toggle are the only structural changes. The saturday-side wire is small: mayor emits `{session_id, phase}` on inject; lucida flips the matching tower's tint via the existing danger-mode path. Tower-as-room maps cleanly to saturday's many-sessions model; eye-track + voice obviates the V0.2 callsign workaround — gaze is the referent. The interesting saturday-original move once the substrate exists: **focus-driven window repositioning** — saturday already knows which session is being addressed, so it can move that terminal forward in 3D space and shift others to peripheral, with the 3D-desktop substrate's transform API. Triggers: (a) lucida's WebXR mode lands and proves the spatial extension reads, (b) the OSS Linux XR stack settles enough to pick a 3D-desktop substrate (WayVR / xrdesktop / Stardust XR / per-app WebXR), (c) saturday's session-targeting confidence is high enough that mis-routes don't yank the wrong window forward. Until all three: voice-first, flat-screen-first; VR remains a research direction, not a default mode.
